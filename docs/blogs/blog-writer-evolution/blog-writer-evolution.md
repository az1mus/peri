# Peri Code: blog-writer skill 如何用负反馈循环稳定输出风格

> **[Peri Code](https://github.com/konghayao/peri)** — 用 Rust 写的开源 Coding Agent，兼容 Claude Code 生态。<https://github.com/KonghaYao/peri>

早期 Peri 的博客文章有一个通病——读完知道它在讲什么，但感受不到人在写。句子工整、段落对称、结论总在最后，内容没有明显错误，但读者读不下去。我们自己重读时也会跳过中间段落，因为每一段都在下判断，却没有一个具体场景把人留住。

根源是风格没有固化成规则。我们每次写博客都靠一轮 prompt（输入给模型的指令）灌输风格偏好，模型这次能按偏好输出，换个题材又恢复默认风格。风格稳定不下来，读者自然觉得 AI 味很重。我们后来把每一轮「这里不对」的反馈都写进 skill（写入 Claude Code 的风格规则文件），形成负反馈循环（把输出偏差回传到规则并修正的闭环）。好的博客风格无法一次写定，只能在每一轮负反馈中修正出来。

## 把每轮修改反馈写成可执行规则

给模型看几篇好文章，模型输出的东西仍然不符合预期。模型输出会复现最显眼的特征——段落对称、金句收尾、emoji 点缀——但抓不到真正让文章好读的东西，比如具体场景、清晰论点、克制的代码块、不绕弯子的句子。

我们把写法从一次性灌输改成逐轮修正。每次写完一篇，指出一处最别扭的地方，把它改写成一条可执行规则，写进 skill。下一轮写作时，这条规则自动生效。

## 一轮反馈只解决一个问题

负反馈循环的形态很简单——写、看、挑一条、改规则、再写。但执行时很容易贪多。一篇文章看完，往往有七八处不顺眼。如果一次塞七八条新规则，skill 会膨胀，模型也难以同时遵循全部规则。我们给自己定了一条规矩——一轮只处理最刺眼的一个问题。

比如早期文章经常出现单句成段。一句话前后空行，孤立在段落中，可能是预告句、提炼句或过渡。读者扫过去会断节奏。我们把它上升为一条明确禁令——正文段落不少于两句，规则类条目除外；所有前后空行的单句都要揪出来，能删就删，删不掉就合并。

这条规则后来被拆得更细，覆盖了预告句、过渡总结、收束提炼三种常见类型。这条规则最初只来自一篇文章里的一个具体问题。

## 把否定描述改成可执行行为

挑出问题只是第一步，更关键的是规则的写法。早期的反馈常常是「这里太 AI 味了」「这段不够自然」。这种描述对模型来说无法量化，每个人的标准也不同。我们把所有反馈都改成具体行为。

「不要写得像 AI」被改写成禁用填充词、总结性空话、反问与设问和模型拟人化，stop_reason（模型返回的停止原因字段）与响应内容不一致这类中性描述被写进规则。每一条规则都具体到可以逐项检查。skill 变成了一组可逐项执行的条件清单。

## 规则按优先级分层

规则多了之后，我们开始给它们分层。第一层管流程——写文章必须走 grill（写作前的质询对齐步骤）、标题、大纲、全文、独立审查、交付这六步。流程规则保证风格问题在落地前被拦截，避免写完再返工。

第二层管结构——开头三段以内、结论前置、每个章节服务核心论点、结尾回应开头。这些规则管文章的结构。

第三层管语言——禁用词、禁用标点、禁用模式、句子结构、小标题写法。这些规则管句子和段落的风格。

分层的好处是，新增规则时有地方放，不会堆成一团。如果一条规则只解决词汇问题，就进禁用词表；如果它管的是写作顺序，就进流程或结构规则。

## 全文之后必须走独立文风审查

skill 成型过程中，最关键的一步是写完全文后派一个独立 subagent（独立审查用的模型实例）做文风审查。这个道理是我们吃亏吃出来的。模型生成的输出很难被同一模型检查——规则已经写进 skill，但生成时的执行有漏洞。独立审查用全新上下文、不受生成时上下文选择的影响，专门检查风格违规，避免走过场。

subagent 要按 SKILL.md 逐项核对，输出问题清单，包含位置、原文摘录、违反规则、修改建议。主 agent 必须据此全部修复，才能交付用户。

这个步骤把负反馈从「写完之后人来看」改成了「写完之后另一个模型按规则检查」。少 AI 味是被规则一遍遍修正出来的。

## 高频规则从反复出现的同类错误中总结

整个 skill 里，有几条规则是从同一类错误的反复出现中总结出来的。单句成段禁令来自早期文章经常把结论单独拎成一行。单独拎成一行会打断阅读流。规则定下来后，所有一句话成段的内容都要处理，要么删，要么并。

小标题必须描述机制，是因为我们曾经用过「字符串匹配，反而是对的」「为 AI 设计工具，直觉是反的」这种标题。读起来像在卖关子。现在的小标题只说这节讲什么行为，比如「所有 ToolResult 收集完再统一写入」「连续失败 5 次，框架注入纠正消息」。这两个都是机制描述型小标题，前者讲 ToolResult（工具调用返回的结果对象）的收集策略，后者讲失败重试的注入逻辑。

代码块数量上限来自另一个观察。以前觉得放代码能显得专业，结果文章被实现细节淹没。现在整篇只放 1-2 个代码块，只用于真实报错或极简示意，其他都用白话讲。

术语首次出现必须带白话解释。我们默认读者懂 chunk（模型返回的一个文本片段）、max_tokens（一次生成最多多少 token）、stop_reason（模型停止生成的标记字段），后来意识到读者的下限是听过方向但没碰过实现。术语第一次出现必须跟一句人话解释。这些规则来自一次次「这里读着不对」的反馈。

## 规则按可执行检查项组织

现在的 blog-writer skill 有十几个章节，从核心原则、写作流程、标题、开头、结构模式、小标题、技术细节、句子结构、语气、禁用词和模式，一直写到结构自检。skill 是给模型用的执行清单，目标是让规则被逐项执行。

小标题这一节直接列出具体禁令——描述态度、写代码、串联多个信息点、用破折号、模糊回指、设问、悬疑故事都在禁止之列。每条都配好坏对比表。

模型生成小标题时，按清单逐项检查。违反一条，就有一处可定位的修改。

Peri 的博客从 AI 默认写作风格主导，变成风格稳定、少 AI 味。靠的是把每一轮不舒服的反馈都固化成规则。规则约束让风格输出稳定，不再随 prompt 波动。
