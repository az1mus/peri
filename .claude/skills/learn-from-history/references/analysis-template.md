# 按天分析模板

General-purpose agent 分析对话历史提取文件时使用的输出格式。

**输入**：由 `scripts/extract_range.py` 生成的按 thread 拆分的目录，
包含 `_index.txt` + 多个 `{thread_id_short}.txt` 文件，每个 <50KB。

**输出**：用 Write 工具写入 `/tmp/learn-summary-YYYY-MM-DD.md`。

---

## 工作流程

1. 先 Read `_index.txt` 了解当天所有 thread 的概况
2. 依次 Read 每个 `.txt` 文件（每个 <50KB，一次读完）
3. `Bash ls /tmp/learn-YYYY-MM-DD/` 确认已读完所有 file
4. Write 分析结果到 `/tmp/learn-summary-YYYY-MM-DD.md`

**重要**：目录中有几个 `.txt` 就要分析几个，不要遗漏。

## 输出格式

```markdown
# 历史分析 — YYYY-MM-DD

**概览:** N 个活跃 thread，M 条消息
**主要目录:** <cwd1> (N threads), <cwd2> (M threads)
**当前项目:** <cwd>（本次分析的目标项目）

---

## 每个 Thread 的条目（按文件顺序，完整覆盖所有 thread）

### Thread N: &lt;用户意图 1-2 句话&gt;
- **Thread ID:** &lt;从文件中提取&gt;
- **项目:** &lt;cwd&gt;
- **消息数:** &lt;N&gt;
- **结果:** 成功 / 部分完成 / 失败 / 被取消
- **关键事件:** 列出失败的工具调用（标注 ✗）、用户纠正（标注[用户纠正]）、异常行为
- **改进线索:** （仅当有异常时）如果 CLAUDE.md 有某条规则能否避免

**重要**：文件中有几个 thread，就要分析几个。不要因为文件大就只分析前几个。
如果分段读取，确保每个 Section 都覆盖到。
```

## 分析指导

### 如何判断异常

重点关注以下信号：

1. **✗ 失败**标记：提取脚本标注了 `✗ 失败` 的工具调用
2. **用户纠正/打断**：用户后续消息中包含否定/纠正/新指示
3. **连续相同工具调用**：提取脚本已标记 `[连续 N 次]`
4. **空结果**：assistant_text 很短紧接着更多工具调用 → 可能在循环
5. **被取消的 thread**：有头无尾，检查有没有丢失消息的迹象

### 如何识别"成功模式"

关注解决了复杂问题的 thread：
- agent 是如何定位问题的？（搜索策略、文件范围）
- agent 是如何验证修复的？（跑测试、Read 确认）
- agent 有没有"多快好省"的操作？比如一次 Grep 找到正确位置而不是逐文件翻

### 如何识别"改进线索"

对每个异常事件，问：
- 如果 CLAUDE.md 里有某条规则，能否避免这个错误？
- 如果某个 skill 有更好的指引，agent 会不会更快？
- 这个错误是系统性问题（LLM/provider bug）还是知识缺失（CLAUDE.md 缺内容）？

### 不要浪费时间在

- 正常完成、无异常的 thread（只记录意图和成功模式，不写"改进线索"）
- 一问一答无工具调用的 thread
- 纯探索性浏览

### 项目分类

- **每 thread 标注项目目录（cwd）**，用于汇总阶段的跨项目分类
- **当前项目 thread**：改进线索应直接建议写入 CLAUDE.md
- **外部项目 thread**：发现放入「外部项目发现」段落，不混入当前项目建议
- **跨项目通用模式**：如 Edit old_string not found 在当前项目和外部项目都出现 → 标注为「项目通用」，建议写入当前项目 CLAUDE.md
- **外部项目特定模式**：如 Slurm 脚本规范（RCS 项目独有）→ 标注项目归属，仅建议写入目标项目
